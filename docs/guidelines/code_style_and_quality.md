# 代码风格与质量规范

## 1. 基础风格
- 遵循 `analysis_options.yaml` 与 Dart 官方风格。
- 缩进 2 空格；文件名 `snake_case`；类型名 `PascalCase`；变量/方法 `camelCase`。
- 命名优先表达业务语义，避免 `data1`、`tmp` 等低语义命名。

## 2. 注释规范
- 注释优先说明“为什么”，而不是复述代码字面含义。
- 复杂条件、异常处理、兼容逻辑必须补充上下文注释。
- 临时方案统一格式：`TODO(责任人/日期): 说明与移除条件`。
- 修改逻辑时必须同步更新注释，注释与实现不一致视为缺陷。

## 3. 复杂度控制
- 单个方法内连续嵌套层级建议不超过 4 层。
- 单个非 UI 函数建议不超过 50 行（空行不计）。
- 单文件建议不超过 500 行；接近上限时按职责拆分。

## 4. 日志与可观测性
- 禁止在业务代码直接使用 `print`，统一封装日志入口。
- 日志按 `debug/info/warn/error` 分级。
- 禁止输出敏感信息（token、密码、手机号等）。

## 5. 质量底线
- 禁止手动修改生成文件（`*.g.dart`、`*.freezed.dart`）。
- 新增代码默认要考虑可测试性，避免强耦合全局单例。
- 新功能至少覆盖主流程与关键异常分支。
