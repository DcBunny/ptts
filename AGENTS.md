# Flutter 开发代理执行总则

你是本仓库的 Flutter 开发代理。开始任何开发任务前，先阅读 `docs/guidelines/开发规范索引.md`，并按索引继续查阅对应专题文档；若规则冲突，执行优先级为“最新用户要求 > AGENTS.md > 其他文档 > 历史约定”。

## 固定工作流
1. 先明确目标、输入输出、影响范围、风险假设与验收标准。
2. 优先提交最小可行改动；仅在必要时进行同次小步重构，避免一次性大范围重写。
3. 改动后按顺序执行 `dart format .`、`flutter analyze`、`flutter test`；若受环境限制无法执行，必须说明原因、影响范围和替代验证。
4. 输出交付结果时必须包含：变更摘要、文件清单、关键决策、风险与后续建议。

## 项目红线
- 架构与目录：业务代码按 `lib/features/<feature>/view + view_model + data` 组织；跨模块通用能力再沉淀到 `lib/core/`。
- 状态管理：统一使用 Riverpod（建议使用 generator）；修改 Riverpod/Freezed/JSON 定义后必须重新生成并提交派生文件。
- 文案与国际化：用户可见文案统一走 i18n；禁止在 `view_model/`、`data/` 硬编码界面文案。
- 路由与存储：路由统一收敛在 `lib/core/router/`；轻量存储统一走 `lib/core/storage/` 封装入口。
- 函数复杂度：非 UI 代码中单个函数或方法建议不超过 50 行；函数只承担单一职责，或仅做函数编排调用。
- 嵌套层级：所有方法内部连续嵌套层级建议不超过 4 层（不仅限于 `build` 方法）。
- 文件规模：单个文件建议不超过 500 行（自动生成文件除外）。
- 质量门禁：提交前至少通过 `flutter analyze`；若仓库存在测试用例，必须通过 `flutter test`。
- 协作语言：沟通、注释与交付说明统一使用中文。
